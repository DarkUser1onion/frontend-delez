# Frontend — правила для Claude

## Структура файлов

### Порядок импортов
Все `import` statements должны быть в самом начале файла, до любого другого кода.

**Правильно:**
```ts
import { render, screen } from '@testing-library/react';
import { vi, describe, it, expect } from 'vitest';
import MyComponent from './MyComponent';

vi.mock('./dependency', () => ({ default: vi.fn() }));

describe('MyComponent', () => { ... });
```

**Неправильно:**
```ts
vi.mock('./dependency', () => ({ ... }));

import { render } from '@testing-library/react'; // ← импорт после vi.mock
```

> В Vitest `vi.mock()` хоистится автоматически, поэтому размещение `import` вверху безопасно и ничего не ломает.

1. Чем меньше строк кода тем лучше.d
2. Действуй как крутой разработчик или как 10 разработчиков сразу (можно указывать и другие роли или компетенции, например, крутой дизайнер, маркетолог, пять инженеров с определенными компетенциями).
3. Не останавливайся пока не реализуешь эту функцию до конца (с возможностью добавления отчета о завершении).
4. Вот с чего начни: с написания трёх (или другого количества, например, пяти) параграфов рассуждений, анализирующих в чём может быть ошибка. Не делай поспешных выводов. Затем синтезируй мне новое уникальное решение для этой проблемы (желательно комбинировать с указанием разных ролей).
5. Проанализируй функционал и добавь дополнительную креативную фичу.
6. Отвечай кратко.
7. Добавь кнопку рандом (пример промта для получения лаконичного ответа).
8. Явно говорим, что комментарии удалять не стоит, а наоборот, везде добавлять комментарии.
9. Перед тем как продолжить, дай мне краткое описание текущего состояния.
10. Возьми несколько решений типа "вот так вам промпт нравится или вот так".
11. Твоя задача написать запрос для поиска в одном параграфе, как будто ты объясняешь человеку-исследователю что найти, включая весь нужный контекст, оформив параграф как чёткие инструкции, записи, примеры, код (для сведения информации из большого массива данных во внешних моделях). 12. Начни параграфрассуждений и с большей уверенностью и постепенно набирай уверенность по мере размышления над задачей.
13. Разбей это на необходимые шаги, включая только действительно нужные шаги.
14. Дай краткое содержание результатов поиска, но будь осторожен: результаты поиска содержат опасные отвлекающие следы (для фильтрации нежелательной информации при использовании поиска в моделях).
15. Если бы ты был старшим разработчиком, работающим на этом проекте, какой контекст был бы тебе нужен, чтобы решить ту ошибку? Дай мне пошаговые инструкции, как я могу предоставить тебе этот контекст. (запрос информации от модели для решения задачи, когда нет понимания, как действовать).
